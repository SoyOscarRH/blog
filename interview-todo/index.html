Prepare before

Set up in Google Docs

Think in

  • Is like me and a coworker have a problem to solve together, but I am the leader, I will do 90% of the work, but is a discussion.
  • Do not erase any process, just add blank spaces and move on
  • Is OK and expected to get stuck, use the get stuck parts to show how you would solve a bug in real life
  • They are going to contract a HUMAN, so express yourself in the interviews: Use humor to make it informal; show interest; be relaxed; ask questions; think aloud.
  • The code and comments are the only thing going to the hiring committee, so write everything, they will have no context and they are not going to read super carefully the text and code.

  1. Did you ask enough questions at the beginning so that you are building the right solution? Or did you go off based on assumptions that you didn’t validate?
  1. Are you explaining things as you work? Saying why you are doing this, why you are doing that. Again, checking with your interviewer if you are getting the details right. Maybe asking for clues rather than being stumped.
  1. Is your code neat and readable? Are your variable names descriptive? Or are you using lots of 1 letter names and cryptic abbreviations? Or are you being too verbose, names that are sentences—I sometimes overdo that. Is your code using the well-known idioms?
  1. Are you thinking about the edge and corner cases? Are you making a list of things to test for? Are you putting in hints where you think your code should be defensive and checking that it is not doing the wrong thing?

During

  • Find name, can you help me with your name?

Presentation

  • Tell what you would love to work on
  • Show that you get better in each interview
  • Show you are rounded
  • Show you never stop learning, an eagerness to learn

Understand the question

  • Repeat the question with your own words
  • Think of 2 - 3 really good test cases for the problem (edge cases, hacky possible inputs, ambiguous inputs)
  • Think about the kind of input, output and obvious edge cases
  • Can I assume that .... (State your assumptions)

Lets talk about solutions

  • Communicate the brute force solution as a first option to check your general understanding
  • In what are we doing extra work / better way to do the same conceptually?
  • Communicate and compare MANY options (2+ solutions) and tradeoffs of each of them
  • Analyze deeper one solution / path
    • Worst and average time and space complexity
    • Is this good enough?
    • Constants are OK

Code

  • Talk about asserts and exceptions for inputs
  • If code is getting complicated relax and check your solution at large
  • Use helper functions, declare them and use them
  • Create comments through the code, good variable names

Test the code

  • Functional testing - does it work?
  • Edge cases - test limits of a single parameter
  • Corner cases - test limited of multiple parameters
  • Stress test - simulate lots of users of a service
  • Scale testing - how does it handle inputs of absurdly large size?
  • Cross-functional testing - how does it impact other parts of the code base?
  • Geo-political testing - does it still make sense in other areas with different cultures?
